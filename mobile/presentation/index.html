The problem

Self-incorporated in Canada means doing the CRA's job

  • Every business expense must map to a T2125 tax category
  • 38 distinct categories across General, Home Office, Vehicle
  • Manual entry: receipts → spreadsheet → year-end accountant scramble
  • Existing apps target consumer finance, not Canadian incorporation

Canadian self-employment / single-shareholder corporations file T2125. That form has a fixed taxonomy — General Business, Home Office, Vehicle. Mint and YNAB don't help: their categories don't map to CRA. Penny owns that pain. ~40s.
Architecture

Hybrid backend: Firestore for CRUD, Next.js for the smart stuff

Two lanes from the Flutter client: (1) Direct FlutterFire to Firestore — used for all CRUD reads/writes because real-time streams + offline cache for free, and Firestore security rules enforce per-user/per-group access. (2) Next.js API over Dio with a Firebase ID token, used only when we need the Gemini API key (server-side), Admin SDK, or multi-doc transactions (e.g. group create = group doc + member doc + activity log). Auth flows through both. ~75s — this is the architectural spine of the talk.
UX · headline feature

Conversational expense entry, not forms

  • Home tab is a chat surface — type or snap a receipt
  • Conversations persist as Firestore subcollections (conversations/{id}/messages)
  • StateNotifierProvider for the active thread; StreamProvider for history
  • Receipt images live in Cloud Storage, URL on the message doc
@riverpod
class Chat extends _$Chat {
  Future<void> send(String text, {File? receipt}) async {
    state = state.copyWith(status: Sending());
    final r = await ref.read(aiRepositoryProvider)
        .analyzeExpense(text: text, image: receipt);
    state = state.copyWith(extracted: r);
  }
}
The Home tab is deliberately a chat UI, not a form-with-fields. People already know how to text. The Riverpod StateNotifier keeps in-flight chat state local; persisted conversations stream from Firestore. ~45s.

Real-time analytics

Dashboard streams Firestore directly into fl_chart

  • StreamProvider on expenses.where(userId).orderBy(date)
  • Aggregations client-side over the stream — no precomputed rollups
  • fl_chart for trend lines + category donut
  • Pull-to-refresh just resubscribes; offline cache fills gaps
Stream<List<ExpenseModel>> personalExpenses(String userId) =>
  _firestore.collection('expenses')
    .where('userId', isEqualTo: userId)
    .where('expenseType', isEqualTo: 'personal')
    .orderBy('date', descending: true)
    .snapshots()
    .map((s) => s.docs.map(ExpenseModel.fromFirestore).toList());
One stream, all the views derived from it. Firestore charges by read, so we read once and pivot in-memory. Works offline because FlutterFire persists the snapshot cache. ~45s.
Domain

CRA T2125 categories — canonical strings, mirrored TS ↔ Dart

Source of truth: src/lib/categories.ts; Dart mirrors the exact strings — written to Firestore as raw labels.

export const expenseCategories = [
  "Advertising (Promotion, gift cards etc.)",
  "Meals and entertainment",  // ... 19 General
  "Home Office - Electricity", // ... 9 Home Office
  "Vehicle - Fuel (gasoline, propane, oil)", // ... 10 Vehicle
] as const;
  • Gemini is prompted with the literal list — no fuzzy mapping
  • Drift between TS and Dart is a release blocker
Why exact strings? Because Firestore queries are string-equality and because reports go straight to an accountant who's looking for these phrases on the T2125 form. Any rename is a data migration. ~55s.
Discipline

Budgets stream the same source; alerts are server-side

  • Per-category monthly limits in budgets_personal / budgets_group
  • Usage = expenses stream filtered by month + category, computed client-side
  • Server checks breach on each POST /api/expenses and writes the notifications doc — client never decides "you went over"
  • FCM push fired from the same code path
The interesting bit is the split: read-side budget math is client-side (cheap, instant), but the "you're over" event is server-authoritative so two devices can't race each other into duplicate alerts. ~45s.
Multi-user

Groups: 4 roles, enforced by 750-line Firestore rules

Roles: owner → admin → member → viewer. Permissions denormalized into each groupMembers doc — rules branch on a single get().

match /groupMembers/{membershipId} {
  allow get: if isAuthenticated() &&
    (resource.data.userId == request.auth.uid ||
     isGroupMember(resource.data.groupId));
  // Server-only — created on group create or invite accept
  allow create: if false;
  allow update: if isGroupOwner(resource.data.groupId) ||
                   request.auth.uid == resource.data.userId;
}
  • allow create: if false — only the API can mint memberships
  • Group creation = one multi-doc txn: group + owner member + activity
Important callout for engineers: Firestore rules enforce mobile-side because mobile uses the client SDK. The Next.js web app uses Admin SDK and bypasses rules — so the rules can't be the only line of defense for server-side flows. That's why critical writes (memberships, notifications) are server-only at the rule layer. ~60s.
